Subject: OSv I/O subsystem improvements -- ZFS, io_uring, multiqueue, and more

Hi Waldek,

I hope you're well. I've been working on a series of improvements to OSv's I/O
subsystem over the past several months and wanted to share what I have so far,
ask for your thoughts, and get your guidance on whether any of this would be
worth upstreaming. I'm happy to rebase, split, rework, or drop anything that
doesn't fit with the project's direction.

Here's a summary of what the branch contains. Everything is on
osv-improvements (rebased on master) and the individual features are also
available on separate feat/* branches.

---

1. OpenZFS 2.4.1 integration (upgrading from in-tree BSD-ZFS)

The biggest change by far. The existing BSD-ZFS code is aging and lacks many
features that OpenZFS 2.x provides (TRIM/DISCARD, encryption, compression
improvements, modern on-disk formats). This series:

  - Replaces the in-tree BSD-ZFS objects with a submodule pointing to the
    upstream OpenZFS 2.4.1 release.
  - Adds an OSv platform layer under external/openzfs/module/os/osv/ that
    bridges OpenZFS's SPL/ZFS kernel module to OSv's bio/device layer.
  - Builds libsolaris.so, zpool.so, and zfs.so as userspace tools.
  - Wires everything into the OSv VFS layer (vfsops/vnops) so that ZFS
    filesystems mount and pass the existing test suite.
  - Implements zfs_vop_cache() for mmap integration (page cache backed by
    ZFS reads, no double-caching risk -- ARC caches compressed blocks, OSv
    page cache caches decompressed 4KB pages for mmap).
  - Supports MAP_SHARED file-backed mmap on ZFS.
  - Implements ftruncate (zfs_freesp and helpers).

I've tried to keep the OSv-side patches minimal and the OpenZFS submodule
untouched, so that future OpenZFS upgrades are straightforward using the
provided scripts/update-openzfs.sh script and patches/ directory.

I'm a bit worried this is too large a change to ask for in one go, so I've
split the ZFS commits into three logical sub-commits (build system, porting
layer, VFS/boot integration) to make review easier. I'd completely understand
if you'd prefer this be held until it's more thoroughly tested, or if you'd
want it broken into a longer series of smaller incremental patches.

2. ZFS encryption integration test

A new test (tests/tst-zfs-encryption.cc) that exercises AES-256-GCM
encryption end-to-end: create an encrypted dataset, write/read data, unload
the key, verify access is blocked, reload the key, verify access is restored.
21/21 test cases pass. This required fixing the libzfs mnttab cache (the
in-memory mount table wasn't being populated on OSv because libzfs_mnttab_add
was guarded behind avl_numnodes > 0 -- fixed in our local copy of libzfs).

3. io_uring support (full Linux 5.10 opcode parity)

Implements the io_uring_setup, io_uring_enter, and io_uring_register syscalls
with complete Linux ABI compatibility. All 40 opcodes are implemented:

  - Read/write: READV, WRITEV, READ, WRITE, READ_FIXED, WRITE_FIXED (with
    registered buffer support).
  - Networking: SENDMSG, RECVMSG, SEND, RECV (with provided buffer group
    selection), ACCEPT, CONNECT, SHUTDOWN.
  - Filesystem: OPENAT, OPENAT2, CLOSE, STATX, FALLOCATE, FADVISE, MADVISE,
    FSYNC, SYNC_FILE_RANGE, RENAMEAT, UNLINKAT, MKDIRAT, SYMLINKAT, LINKAT.
  - Polling: POLL_ADD (synchronous), POLL_REMOVE, EPOLL_CTL.
  - Control: NOP, TIMEOUT, TIMEOUT_REMOVE, LINK_TIMEOUT, ASYNC_CANCEL.
  - Zero-copy: SPLICE (via bounce-copy read+write), TEE.
  - Buffer management: PROVIDE_BUFFERS, REMOVE_BUFFERS.
  - File table: FILES_UPDATE.

Registration opcodes implemented: IORING_REGISTER_BUFFERS/FILES/EVENTFD/
PROBE/PBUF_RING/SYNC_CANCEL and their UNREGISTER counterparts.

Linked request chains (IOSQE_IO_LINK, IOSQE_IO_HARDLINK), IO drain
(IOSQE_IO_DRAIN), async cancel (ASYNC_CANCEL), and provided buffer groups
are all implemented. Includes large-page (2MB) ring buffer mapping using
hw_ptep<1> for efficient ring management at high queue depths.

In OSv's single-address-space model, "async" means dispatching each SQE
chain to a sched::thread; the syscall overhead (~50-100 ns) is low enough
that io_uring's primary benefit on OSv is batching and linked-request
semantics rather than user/kernel boundary avoidance.

4. MADV_HUGEPAGE and MAP_HUGETLB

OSv already uses 2MB huge pages by default for anonymous mappings.
MADV_HUGEPAGE re-enables huge pages after MADV_NOHUGEPAGE (clears the
mmap_small flag). MAP_HUGETLB requests strict huge-page-only allocation
(returns ENOMEM if huge pages are unavailable rather than falling back to
4KB). Tests are included.

5. musl libc upgrade: 1.1.24 -> 1.2.1

Updates the musl submodule. The main user-visible change is proper large-file
support (off_t is always 64-bit in 1.2.x). A small fix to
libc/stdio/__string_read.c was needed to adapt to the new musl internals.

6. VirtIO-BLK multiqueue (per-CPU queue dispatch)

Replaces the single global _lock mutex with per-queue locks and CPU-affine
queue selection (sched::cpu::current()->id % _num_queues). On a multiqueue-
capable hypervisor (VIRTIO_BLK_F_MQ), multiple CPUs can now submit I/O
simultaneously without serializing on a single global lock. Completion
handling creates one thread per queue; the queue-0 thread is interrupt-driven
and also drains other queues when woken.

A concurrency bug was also fixed in the drain path: the drain-all loop
(qid==0) and the per-queue poll path (qid>0) were both missing
_queue_locks[q] around their ring-buffer walk, which could race with request
completion and corrupt the used-ring index under high queue depth.

7. Pagecache improvements: sequential readahead, periodic writeback, and
   filesystem-signaled writeback API

Sequential readahead: after fetching a page on an mmap fault miss, the
pagecache layer checks whether the immediately preceding page is cached (a
signal that access is sequential). If so, it speculatively prefetches the
next 4 pages before returning, hiding one round-trip of I/O latency on
sequential scans.

Periodic writeback: a background thread (pagecache-wb) wakes every N seconds
(default 5, tunable via pagecache_wb_interval_secs) and flushes all dirty
write-cached pages. Previously, dirty pages were only flushed on LRU eviction
or explicit fsync.

Filesystem-signaled writeback API: pagecache::writeback_inode(dev, ino,
start, end) lets filesystem VOP callbacks (VOP_FSYNC, checkpoint hooks,
pre-unmount) request writeback of specific inode ranges without holding a
vfs_file*. A C-linkage entry point osv_pagecache_writeback_inode() is
provided for C filesystem code. pagecache::writeback_all() is available for
system-wide sync at shutdown.

Three correctness fixes:
  - Fixed thread-unsafe static std::stack<> in pagecache::sync() (now a
    local vector per call).
  - Added a phase that promotes PTE-dirty bits to the software _dirty flag
    before collecting pages, so pagecache::sync() no longer misses MAP_SHARED
    writes that haven't yet been unmapped.
  - Fixed sys_fsync() in fs/vfs/vfs_syscalls.cc to flush the write cache
    (via osv_pagecache_writeback_inode) before calling VOP_FSYNC.
    Previously, MAP_SHARED mmap writes that had not been msync()ed would
    not reach the filesystem's journal even after fsync() returned.

8. Crucible block device driver (Oxide Computer hardware)

A VirtIO-like block device driver for Oxide Computer's Crucible distributed
block storage. This is Oxide-specific hardware not present in standard QEMU,
so the driver is opt-in (not compiled into the default 'all' profile). Use
conf_drivers_profile=crucible to include it. I've added a note to this effect
in the driver source.

9. Various build and test improvements

  - Nix flake updated to replace boost177 with the current Boost.
    httpserver-api updated for Boost 1.74+/1.78+ API changes (io_context,
    mutable_buffer, std::isinf). modules/common.gmk now searches for
    libboost_filesystem.so and accepts a boost_lib_dir override for NixOS
    where headers and libs live in separate store paths.
  - NixOS build fixes: libz.so and OpenSSL 3.x discovery via LD_LIBRARY_PATH
    fallback; manifest_from_host.sh filters lddtree "=> None" lines.
  - Filesystem benchmark (tests/tst-fs-bench.cc) measures sequential and
    random I/O, metadata ops, and mmap throughput across all filesystems.
  - ZFS direct I/O test (tests/tst-zfs-direct-io.cc).
  - ZFS recordsize throughput benchmark (tests/tst-zfs-recordsize.cc).
  - ZFS database simulation benchmark (tests/tst-zfs-db-sim.cc): simulates
    a Postgres WAL workload (8 KiB random pages, 750 MiB dataset) across
    five dataset configurations. Demonstrates that O_DIRECT + primarycache=
    metadata + lz4 achieves ~1100 txn/s on 128 MiB RAM and ~1450 txn/s on
    512 MiB, with configurations that would OOM automatically skipped.
  - Shared memory consistency test (tests/tst-shm-consistency.cc).

Honest assessment of what still needs work:

  - The OpenZFS integration is functional but has not been stress-tested at
    the level I'd want before calling it production-ready. I'd particularly
    appreciate your thoughts on the ARC/pagecache layering (documented in
    core/pagecache.cc) and whether the current "route ZFS through read_cache
    via zfs_vop_cache()" approach is the right long-term design.

  - Several low-level stability fixes were needed to make ZFS run reliably
    at 128 MiB RAM: (a) virt_to_phys() now walks the page table for VMA-
    mapped DMA buffers that land below phys_mem rather than asserting, since
    ZFS malloc_large() buffers fragment physical memory under sustained I/O;
    (b) kthread stacks were increased from 16 KiB to 256 KiB to prevent ZFS
    thread overflow; (c) the ARC shrinker is registered inside
    register_shrinker_arc_funs() (called after arc_init()) rather than in
    bsd_shrinker_init() which runs before libsolaris.so is loaded; (d) the
    ARC's arc_default_max() uses a tiered formula capping the cache at 1/8
    of RAM for 128–256 MiB systems to leave room for the ZIO pipeline.
    These are documented in the commit messages and in the code.

  - The io_uring implementation is complete with respect to the Linux 5.10
    ABI. On OSv there is no real async I/O kernel infrastructure, so all
    opcodes are dispatched to sched::threads. The primary value is ABI
    compatibility (existing io_uring userspace libraries work unchanged) and
    batching semantics. SQPOLL mode is implemented as well.

  - Periodic writeback uses a 5-second default interval. Linux defaults to
    30 seconds (dirty_writeback_centisecs=1500) but OSv's simpler memory
    model might benefit from a shorter interval. The interval is tunable at
    runtime via the pagecache_wb_interval_secs global.

I've tried to structure the commits so that each is logically independent and
builds cleanly on its own. If it would be easier to review individual features
as separate PRs, I'm happy to do that instead.
